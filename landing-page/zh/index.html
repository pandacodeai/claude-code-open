重磅功能 2026年2月

Web UI：从终端走向浏览器，打造全栈 AI IDE

为什么需要 Web UI？

CLI 是开发者的利器，但它有天然的局限：无法展示富文本、不支持多窗口布局、缺少可视化调试能力。我们的目标是：在浏览器中获得和 VSCode 一样流畅的 AI 编程体验，同时保留 CLI 的全部能力。

技术架构

Web UI 采用经典的三层架构，每一层都经过精心选型：

  • 客户端 — React 18 + Vite + TypeScript。多页路由支持 AuthPage（OAuth 登录）、BlueprintPage（工作流编辑器）、CodeBrowserPage（代码浏览）、SwarmConsole（多智能体控制台）
  • 通信层 — WebSocket 实时双向消息 + tRPC 类型安全 API。消息流、工具调用状态、Agent 进度全部实时推送
  • 服务端 — Express.js + Better-SQLite3。SessionManager 管理会话持久化，PermissionHandler 处理权限控制，TaskManager 协调后台任务，TerminalManager 对接 xterm.js 终端模拟

核心组件一览

Web UI 包含 30+ 精心设计的 React 组件：

  • 消息系统 — MarkdownContent 渲染器、ToolCall 可视化、PermissionDialog 权限确认、UserQuestionDialog 交互式问答
  • 代码浏览器 — 虚拟化 TreeView 文件树、语法高亮、调用图可视化
  • 配置面板 — Hooks 配置、权限管理、API 设置、缓存控制、MCP 服务器管理、插件管理
  • 终端集成 — 基于 xterm.js 的完整终端模拟，可在 Web 中直接执行命令

OAuth 认证流程

我们实现了完整的 OAuth 认证体系：用户通过 AuthPage 登录后，OAuthManager 管理 token 生命周期，自动刷新、多设备同步。这意味着你可以在任何有浏览器的设备上安全地使用 Claude 编程助手。

亮点数据：30+ React 组件 · WebSocket 实时通信 · 4 个独立页面 · xterm.js 终端 · SQLite 持久化 · OAuth 全流程

多智能体 2026年1月

Agent Teams：当一个 AI 不够用，就让一群 AI 协作

单一 Agent 的瓶颈

单个 Claude 实例处理复杂任务时，面临上下文窗口限制、专注力分散、无法并行等问题。我们的答案是：让多个 Agent 组成团队，每个成员专注自己的职责，通过消息系统协作完成大型任务。

架构设计

Agent Teams 系统由四个核心工具组成：

TeamCreate — 创建团队，定义团队目标和成员角色
TeamSendMessage — 团队成员间发送消息，共享发现和进度
Teammate — v2.1.32+ 新增，单个队友的上下文管理
TeamDelete — 任务完成后清理团队资源

消息通信机制

每个 Agent 拥有独立的上下文窗口和工具权限，通过结构化消息进行通信。TeamSendMessage 支持定向消息（发送给特定成员）和广播消息（全队共享），确保信息在正确的时间到达正确的人。

上下文记忆共享

团队成员通过 Memory 系统共享关键发现。当一个 Agent 分析出代码的关键架构信息，它可以将这些知识持久化到共享记忆中，其他成员无需重复劳动即可获取这些洞察。

典型使用场景

  • 大型重构 — 3 个 Agent 分别负责前端、后端、测试，并行重构同一系统
  • 代码审查 — 安全审查员 + 性能审查员 + 架构审查员各司其职
  • 文档生成 — 一个 Agent 分析代码，一个编写文档，一个生成示例

核心能力：团队创建/删除 · 成员角色定义 · 定向/广播消息 · 共享记忆 · 并行执行 · 进度监控

工作流引擎 2026年1月

Blueprint：用可视化工作流编排多智能体任务

从手动调度到自动编排

Agent Teams 让多智能体协作成为可能，但手动管理团队分工仍然繁琐。Blueprint 系统在此基础上更进一步：你描述目标，系统自动生成执行计划、分配任务、调度 Worker、验证结果。

七大核心工具

  • Blueprint — 定义项目蓝图，描述最终目标和约束条件
  • GenerateBlueprint — AI 自动分析蓝图，生成可执行的任务分解
  • GenerateDesign — 根据蓝图生成详细的技术设计方案
  • StartLeadAgent — 启动主控 Agent，负责全局调度和决策
  • UpdateTaskPlan — 动态更新任务计划，响应执行中的变化
  • DispatchWorker — 分派 Worker Agent 执行具体子任务
  • TriggerE2ETest — 触发端到端测试，验证整体交付质量

工作流执行过程

用户描述目标 → GenerateBlueprint 生成计划
→ StartLeadAgent 启动主控
→ DispatchWorker 分派任务到各 Worker
→ Worker 并行执行，通过消息汇报进度
→ Lead Agent 动态调整 (UpdateTaskPlan)
→ TriggerE2ETest 验证最终结果

Web UI 中的 Blueprint 编辑器

在 BlueprintPage 中，你可以可视化地查看整个工作流的执行状态：哪些 Worker 在运行、任务进度如何、哪里出现了阻塞。Swarm Console 提供了 Agent Explorer、任务树、Worker 面板等丰富的监控工具。

自动化链路：目标描述 → 蓝图生成 → 设计方案 → Lead Agent → Worker 分派 → E2E 验证 —— 全流程自动化

架构设计 2026年1月

插件 & Hooks：用 3 个设计模式打造无限可扩展的工具系统

为什么可扩展性至关重要

AI 编程工具的生命力在于生态。一个封闭的系统终将被淘汰，而一个开放的平台可以汇聚社区的力量不断进化。我们从第一天就把可扩展性作为核心设计目标。

三大设计模式

1. Registry Pattern — ToolRegistry

所有工具通过 ToolRegistry 动态注册和管理。每个工具继承 BaseTool，定义 Zod schema 输入校验和 execute() 方法。支持运行时添加/移除工具，allow/disallow 列表控制工具可见性。

2. Plugin Pattern — PluginManager

插件系统提供完整的生命周期管理：加载 → 初始化 → 激活 → 停用 → 卸载。插件可以注册新工具、修改现有行为、添加 UI 组件。插件目录支持全局（~/.claude/plugins/）和项目级（.claude/plugins/）。

3. Observer Pattern — Hooks System

Hooks 系统让你在任何工具执行前后注入自定义逻辑。Pre-hook 可以拦截、修改甚至取消工具调用，Post-hook 可以处理结果、记录审计日志。这是实现安全审计、自定义审批流的基础设施。

Skill 系统 — 自定义命令

Skill 系统允许创建可复用的命令模板，存放在 ~/.claude/skills/ 或 .claude/commands/ 目录。一条 /commit 斜杠命令背后，是一个完整的提交工作流模板。

扩展方式：ToolRegistry 动态注册 · PluginManager 生命周期 · Hooks pre/post 钩子 · Skill 自定义命令 · MCP 外部协议

国际化 2026年2月

i18n：为什么我们没用现成框架，而是从零搭建国际化系统

现有方案的问题

i18next、react-intl 等成熟框架功能强大，但对我们的场景来说太重了。我们需要的是：CLI 和 Web UI 共用一套翻译、零运行时依赖、支持动态加载、Tree-shaking 友好。

自研方案的核心设计

  • 统一翻译文件 — 中英文翻译以 JSON 格式存放在 src/i18n/ 目录，CLI 和 Web UI 共用同一份翻译源
  • 类型安全 — TypeScript 类型推导确保每个翻译 key 都有对应值，编译期就能发现缺失翻译
  • 按需加载 — 语言包支持动态 import，只加载当前语言，减小 bundle 体积
  • 落地页双语 — 两个独立 HTML 文件 + 语言切换链接，零 JS 开销实现多语言落地页

覆盖范围

国际化不只是 UI 文本翻译。我们覆盖了：CLI 提示语、错误信息、工具描述、Web UI 界面、落地页、配置说明 —— 确保中文用户获得完全原生的体验。

设计原则：零依赖 · 类型安全 · CLI/Web 共用 · 按需加载 · 编译期校验

深度解析 2026年2月

攻克 Claude Code OAuth 代理之谜：如何修复「凭证仅限 Claude Code 使用」错误

问题背景

在构建透明代理以跨设备共享 Claude 订阅时，我们遇到了一个令人困惑的 400 错误：

"This credential is only authorized for use with Claude Code and cannot be used for other API requests."

通过 Postman 发送简单请求可以正常工作，但转发真实的 Claude Code CLI 请求却失败了。到底哪里不同？

问题根因

经过数小时的抓包分析和混淆代码逆向，我们发现了根本原因：代理破坏了 system prompt 的 block 结构。

官方 Claude Code 发送的 system prompt 是一个包含特定结构的 text block 数组：

Block 0: "x-anthropic-billing-header: cc_version=2.1.31.593; cc_entrypoint=cli;"
Block 1: "You are Claude Code, Anthropic's official CLI for Claude..."
Block 2: "...其他系统提示词..."

x-anthropic-billing-header 是 Anthropic 用于验证 Claude Code 客户端身份的特殊标记块。我们代理的身份注入逻辑找到第一个 text block（billing header），然后把 Claude Code 身份标识拼接到它前面：

// BUG: 这破坏了 billing header block！
block.text = CLAUDE_CODE_IDENTITY + '\n\n' + block.text;

结果：billing header 不再位于任何 block 的开头，API 找不到它就拒绝了请求。

解决方案

我们更新了身份注入逻辑：

  1. 跳过 billing header block - 永远不修改以 x-anthropic-billing-header 开头的 block
  2. 将身份标识作为独立 block 插入 - 如果没有现有身份，在 billing header 之后插入新 block，保持官方结构
  3. 移除 prompt-caching-scope beta - v4 转发路径缺少这个关键的 beta header 过滤
// FIX: 跳过 billing block，在其后插入身份标识
const isBillingBlock = (b) => b.text?.startsWith('x-anthropic-billing-header');
const insertIdx = system.findIndex(isBillingBlock) + 1;
system.splice(insertIdx, 0, { type: 'text', text: IDENTITY });

核心收获

  • Block 结构很重要 - Claude Code 的 system prompt 不只是文本，而是每个 block 都有语义含义的结构化数组
  • Billing header 神圣不可侵犯 - x-anthropic-billing-header block 用于客户端验证，必须保持原样
  • Beta header 需要谨慎处理 - prompt-caching-scope 会改变缓存行为，在代理 OAuth 请求时必须移除

最终效果：代理现在可以正确转发 OAuth 订阅请求，让你能够在多台设备之间无缝共享 Claude 订阅。
