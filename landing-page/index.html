Major Feature February 2026

Web UI: From Terminal to Browser — Building a Full-Stack AI IDE

Why Web UI?

CLI is a developer's power tool, but it has inherent limitations: no rich text, no multi-panel layouts, no visual debugging. Our goal: deliver the full AI coding experience in a browser while retaining every CLI capability.

Technical Architecture

The Web UI follows a classic three-layer architecture, each layer carefully chosen:

  • Client — React 18 + Vite + TypeScript. Multi-page routing: AuthPage (OAuth login), BlueprintPage (workflow editor), CodeBrowserPage (code exploration), SwarmConsole (multi-agent dashboard)
  • Communication — WebSocket real-time bidirectional messaging + tRPC type-safe API. Message streams, tool call status, and agent progress all pushed in real-time
  • Server — Express.js + Better-SQLite3. SessionManager for persistence, PermissionHandler for access control, TaskManager for background coordination, TerminalManager for xterm.js integration

Core Components at a Glance

The Web UI includes 30+ carefully designed React components:

  • Messaging System — MarkdownContent renderer, ToolCall visualization, PermissionDialog, UserQuestionDialog
  • Code Browser — Virtualized TreeView file explorer, syntax highlighting, call graph visualization
  • Configuration Panels — Hooks config, permissions management, API settings, cache control, MCP server management, plugin management
  • Terminal Integration — Full terminal emulation via xterm.js, execute commands directly in the browser

OAuth Authentication Flow

We implemented a complete OAuth authentication system: users log in via AuthPage, OAuthManager handles token lifecycle with auto-refresh and multi-device sync. This means you can securely use the Claude coding assistant on any device with a browser.

Key Numbers: 30+ React components · WebSocket real-time · 4 independent pages · xterm.js terminal · SQLite persistence · Full OAuth flow

Multi-Agent January 2026

Agent Teams: When One AI Isn't Enough, Make Them Collaborate

The Single-Agent Bottleneck

A single Claude instance faces context window limits, attention fragmentation, and inability to parallelize. Our answer: multiple agents forming teams, each focused on their role, collaborating through a messaging system to tackle large tasks.

Architecture Design

The Agent Teams system consists of four core tools:

TeamCreate — Create teams, define goals and member roles
TeamSendMessage — Send messages between members, share findings
Teammate — v2.1.32+ individual teammate context management
TeamDelete — Clean up team resources after task completion

Messaging Mechanism

Each agent has an independent context window and tool permissions, communicating through structured messages. TeamSendMessage supports directed messages (to specific members) and broadcasts (shared with the whole team).

Shared Context Memory

Team members share key discoveries through the Memory system. When one agent uncovers critical architecture insights, it persists them to shared memory — other members access these insights without redundant work.

Typical Use Cases

  • Large Refactoring — 3 agents handling frontend, backend, and testing in parallel
  • Code Review — Security reviewer + performance reviewer + architecture reviewer, each with their specialty
  • Documentation — One agent analyzes code, one writes docs, one generates examples

Core Capabilities: Team create/delete · Role definition · Directed/broadcast messaging · Shared memory · Parallel execution · Progress monitoring

Blueprint: Orchestrating Multi-Agent Tasks with Visual Workflows

From Manual Dispatch to Automated Orchestration

Agent Teams enable multi-agent collaboration, but manual team management is still tedious. Blueprint takes it further: you describe the goal, the system auto-generates execution plans, assigns tasks, dispatches workers, and validates results.

Seven Core Tools

  • Blueprint — Define project blueprints with goals and constraints
  • GenerateBlueprint — AI analyzes blueprints, generates executable task breakdowns
  • GenerateDesign — Generate detailed technical design documents from blueprints
  • StartLeadAgent — Launch the lead agent for global scheduling and decisions
  • UpdateTaskPlan — Dynamically update task plans in response to execution changes
  • DispatchWorker — Dispatch worker agents for specific subtasks
  • TriggerE2ETest — Trigger end-to-end tests to validate overall delivery quality

Workflow Execution Flow

User describes goal → GenerateBlueprint creates plan
→ StartLeadAgent launches coordinator
→ DispatchWorker assigns tasks to workers
→ Workers execute in parallel, report progress
→ Lead Agent dynamically adjusts (UpdateTaskPlan)
→ TriggerE2ETest validates final results

Blueprint Editor in Web UI

In the BlueprintPage, you can visually monitor the entire workflow: which workers are running, task progress, and where bottlenecks occur. The Swarm Console provides Agent Explorer, task tree, worker panels, and rich monitoring tools.

Automation Pipeline: Goal description → Blueprint generation → Design docs → Lead Agent → Worker dispatch → E2E validation — fully automated

Architecture January 2026

Plugin & Hooks: Building an Infinitely Extensible Tool System with 3 Design Patterns

Why Extensibility Matters

An AI coding tool's longevity lies in its ecosystem. A closed system will eventually be outgrown, while an open platform harnesses community power to continuously evolve. We made extensibility a core design goal from day one.

Three Design Patterns

1. Registry Pattern — ToolRegistry

All tools register dynamically through ToolRegistry. Each extends BaseTool, defines Zod schema input validation and an execute() method. Supports runtime add/remove and allow/disallow lists for visibility control.

2. Plugin Pattern — PluginManager

The plugin system provides full lifecycle management: load → init → activate → deactivate → unload. Plugins can register new tools, modify existing behavior, and add UI components. Plugin directories support global (~/.claude/plugins/) and project-level (.claude/plugins/).

3. Observer Pattern — Hooks System

The Hooks system lets you inject custom logic before and after any tool execution. Pre-hooks can intercept, modify, or cancel tool calls. Post-hooks can process results and log audits. This is the foundation for security auditing and custom approval workflows.

Skill System — Custom Commands

The Skill system allows creating reusable command templates stored in ~/.claude/skills/ or .claude/commands/. Behind a simple /commit slash command lies a complete commit workflow template.

Extension Points: ToolRegistry dynamic registration · PluginManager lifecycle · Hooks pre/post · Skill custom commands · MCP external protocol

Internationalization February 2026

i18n: Why We Built Our Own Instead of Using an Existing Framework

Problems with Existing Solutions

i18next, react-intl, and other mature frameworks are powerful but too heavy for our use case. We needed: shared translations between CLI and Web UI, zero runtime dependencies, dynamic loading support, and tree-shaking friendliness.

Core Design of Our Custom Solution

  • Unified Translation Files — Translations stored as JSON in src/i18n/, shared between CLI and Web UI
  • Type Safety — TypeScript inference ensures every translation key has a corresponding value, catching missing translations at compile time
  • On-demand Loading — Language packs support dynamic import, loading only the current language to reduce bundle size
  • Bilingual Landing Page — Two independent HTML files + language switch links, zero JS overhead for multi-language landing pages

Coverage

Internationalization goes beyond UI text. We cover: CLI prompts, error messages, tool descriptions, Web UI interface, landing pages, and configuration docs — ensuring Chinese users get a completely native experience.

Design Principles: Zero dependencies · Type-safe · CLI/Web shared · On-demand loading · Compile-time validation

Deep Dive February 2026

Cracking the Claude Code OAuth Proxy Mystery: How We Fixed the "Credential Only Authorized for Claude Code" Error

The Problem

When building a transparent proxy to share Claude subscriptions across devices, we encountered a frustrating 400 error:

"This credential is only authorized for use with Claude Code and cannot be used for other API requests."

The proxy worked perfectly for simple requests via Postman, but failed when forwarding actual Claude Code CLI requests. What was different?

The Investigation

After hours of packet analysis and code diving into the minified cli.js, we discovered the root cause: the proxy was corrupting the system prompt block structure.

Official Claude Code sends system prompts as an array of text blocks with a specific structure:

Block 0: "x-anthropic-billing-header: cc_version=2.1.31.593; cc_entrypoint=cli;"
Block 1: "You are Claude Code, Anthropic's official CLI for Claude..."
Block 2: "...rest of system prompt..."

The x-anthropic-billing-header is a special marker block that Anthropic uses to validate Claude Code client identity. Our proxy's identity injection logic found the first text block (the billing header) and prepended the Claude Code identity to it:

// BUG: This corrupted the billing header block!
block.text = CLAUDE_CODE_IDENTITY + '\n\n' + block.text;

Result: The billing header was no longer at the start of any block, so the API couldn't find it and rejected the request.

The Fix

We updated the identity injection logic to:

  1. Skip billing header blocks - Never modify blocks starting with x-anthropic-billing-header
  2. Insert identity as a separate block - If no identity exists, insert it AFTER the billing header, maintaining the official structure
  3. Remove prompt-caching-scope beta - The v4 forward path was missing this critical beta header filtering
// FIX: Skip billing blocks, insert identity after them
const isBillingBlock = (b) => b.text?.startsWith('x-anthropic-billing-header');
const insertIdx = system.findIndex(isBillingBlock) + 1;
system.splice(insertIdx, 0, { type: 'text', text: IDENTITY });

Key Takeaways

  • Block structure matters - Claude Code's system prompt isn't just text; it's a structured array where each block has semantic meaning
  • Billing headers are sacred - The x-anthropic-billing-header block is used for client validation and must remain untouched
  • Beta headers need careful handling - prompt-caching-scope changes caching behavior and must be removed when proxying OAuth requests

Result: The proxy now correctly forwards OAuth subscription requests, allowing you to share your Claude subscription across multiple devices seamlessly.
